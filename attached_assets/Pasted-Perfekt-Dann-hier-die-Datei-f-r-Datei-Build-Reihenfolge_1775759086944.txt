Replit Build-Reihenfolge

Ziel:

CreatorOS = Plattform
ContentOS = erstes echtes Modul

⸻

Step 1 — zentrale Modul-Definition bauen

Datei:

src/config/modules.ts

Zweck:

Eine einzige Quelle der Wahrheit für alle Module.

Soll rein:
	•	id
	•	name
	•	status
	•	category
	•	description
	•	shortDescription
	•	routes
	•	isCore
	•	optional available

Beispielstruktur:

export type CreatorOSModule = {
  id: string;
  name: string;
  status: "active" | "planned" | "beta";
  category: "core" | "addon";
  shortDescription: string;
  description: string;
  isCore: boolean;
  routes: {
    overview: string;
    app?: string;
  };
};

export const modules: CreatorOSModule[] = [
  {
    id: "contentos",
    name: "ContentOS",
    status: "active",
    category: "core",
    shortDescription: "Plan, generate and manage content workflows.",
    description: "ContentOS is the content workflow module inside CreatorOS.",
    isCore: true,
    routes: {
      overview: "/modules/contentos",
      app: "/modules/contentos/app",
    },
  },
  {
    id: "brandos",
    name: "BrandOS",
    status: "planned",
    category: "core",
    shortDescription: "Define brand foundations, positioning and messaging.",
    description: "BrandOS is the brand strategy module inside CreatorOS.",
    isCore: true,
    routes: {
      overview: "/modules/brandos",
    },
  },
];

Done, wenn:
	•	du Module nicht mehr hart in UI reinschreibst
	•	die Module Page daraus rendern kann

⸻

Step 2 — ContentOS Modul-spezifische Daten trennen

Datei:

src/data/contentos.ts

Zweck:

ContentOS-Inhalte nicht direkt in Components hardcoden.

Soll rein:
	•	Hero Content
	•	Features
	•	Workflow Steps
	•	Use Cases
	•	Tabs / Sections

Beispiel:

export const contentOSData = {

Done, wenn:
	•	ContentOS Seite datengetrieben wird
	•	du später andere Module gleich bauen kannst

Step 4 — gemeinsames Modul-Seitenlayout bauen

Datei:

src/components/modules/ModulePageLayout.tsx

Zweck:

Wiederverwendbare Struktur für jede Modul-Detailseite.

Soll enthalten:
	•	Hero
	•	optional Feature Section
	•	optional Workflow
	•	optional CTA
	•	children slot für modul-spezifische Abschnitte

Vorteil:

Nicht jedes Modul neu zusammenkleben.

⸻

Step 5 — Modules Overview Page auf Registry umstellen

Datei:

src/pages/Modules.tsx

Zweck:

Die Übersicht soll aus modules.ts gerendert werden.

Soll tun:
	•	modules importieren
	•	jede Modul-Karte daraus bauen
	•	Status anzeigen
	•	auf routes.overview verlinken

Done, wenn:
	•	neue Module nur in modules.ts angelegt werden müssen
	•	die Overview automatisch wächst

⸻

Step 6 — ContentOS Modulseite bauen

Datei:

src/modules/contentos/ContentOSPage.tsx

Zweck:

Die Modul-Detailseite für ContentOS.

Aufbau:
	•	ModulePageLayout
	•	Daten aus contentos.ts
	•	CTA → /modules/contentos/app

Enthalten:
	•	Hero
	•	Features
	•	Workflow
	•	evtl. Use Cases
	•	CTA

Done, wenn:
	•	ContentOS wie ein echtes Plattform-Modul wirkt
	•	nicht wie eine lose Unterseite

⸻

Step 7 — ContentOS App Entry Screen bauen

Datei:

src/modules/contentos/ContentOSApp.tsx

Zweck:

Die eigentliche Modul-App-Oberfläche für v1.

Für jetzt einfach halten:

Oben:
	•	Modultitel
	•	Tabs / Toggle

Inhalt:
	•	Overview
	•	Generate
	•	Library

MVP:

Noch keine echte Backend-Logik.
Nur saubere Strukturen und Dummy-State.

⸻

Step 8 — App-interne Shared Komponenten

Ordner:

src/components/modules/contentos/

Dateien optional:
	•	ContentOSTabs.tsx
	•	ContentOSEmptyState.tsx
	•	ContentOSOverviewPanel.tsx

Wenn es klein bleibt, kann es auch alles in ContentOSApp.tsx leben.
Nicht künstlich over-engineeren.

⸻

Step 9 — Routing finalisieren

Datei:

je nachdem Projektstruktur, vermutlich:
	•	src/App.tsx
	•	oder src/main.tsx
	•	oder eigener router file

Routen ergänzen:

<Route path="/modules" element={<Modules />} />
<Route path="/modules/contentos" element={<ContentOSPage />} />
<Route path="/modules/contentos/app" element={<ContentOSApp />} />

Später dann:

<Route path="/modules/brandos" element={<BrandOSPage />} />

Definition of Done pro Step

Step 1 fertig, wenn:
	•	Module nicht mehr in UI hardcoded sind

Step 5 fertig, wenn:
	•	Modules Page komplett aus Registry kommt

Step 6 fertig, wenn:
	•	ContentOS eine glaubwürdige Modul-Detailseite hat

Step 7 fertig, wenn:
	•	ContentOS App Entry sauber zwischen Overview / Generate / Library trennt

Gesamt fertig, wenn:
	•	CreatorOS sich wie Plattform anfühlt
	•	ContentOS sich wie Modul anfühlt
	•	weitere Module einfach ergänzt werden können

Wichtige Warnung

	•	Backend

	•	Billing
	•	AI orchestration
	•	DB sync
	•	Notion sync
	•	Automations

Erst die Plattform-Semantik sauber machen.
Sonst baust du Technik auf begrifflichem Nebel.

Der beste Prompt für Replit

Den würde ich jetzt wirklich so schicken:

Refactor the current CreatorOS project into a platform-first modular architecture.

Core rule:
- CreatorOS = platform
- ContentOS = module

Please implement this in the following order:

1. Create a typed module registry in `src/config/modules.ts`
2. Move ContentOS-specific copy/data into `src/data/contentos.ts`
3. Build reusable shared module UI components:
   - ModuleHero
   - ModuleFeatureGrid
   - ModuleWorkflow
   - ModuleCTA
   - ModulePageLayout
4. Refactor the Modules page to render from the module registry
5. Build `ContentOSPage.tsx` as the first proper module detail page
6. Build `ContentOSApp.tsx` with a clean internal structure for:
   - Overview
   - Generate
   - Library
7. Wire routing for:
   - /modules
   - /modules/contentos
   - /modules/contentos/app
8. Keep styling aligned with the existing premium clean design system
9. Use mock data only
10. Do not add backend or billing logic

Goal:
Make CreatorOS feel like a real modular platform and ContentOS feel like the first real module.
