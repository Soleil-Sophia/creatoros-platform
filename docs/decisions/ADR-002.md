## 5. Context

### Background

At the time this ADR was written, the CreatorOS ecosystem consisted of three repositories:

| Repository | Remote | Description |
|---|---|---|
| `creatoros-platform` | `soleil-sophia/creatoros-platform` | Main platform — React 18 + React Router 7 + Vite + Supabase |
| `creatoros-insight` | (Lovable-origin) | Frontend prototype with additional modules and extended BrandProfile |
| `content-fabrik` | (separate origin) | Node/Express content generation service; stub AI; API contract reference |

These repositories diverged over time. `creatoros-insight` contains production-worthy capabilities (LaunchOS, ManagementOS, AnalyticsOS, extended BrandOS wizard, diagnostics) that do not exist in `creatoros-platform`. `content-fabrik` contains the ContentOS API contract and a stub implementation, but no verified AI runtime.

### Trigger

A consolidation analysis (CR-001, completed 2026-07-16) identified:

- 6 structural conflicts (C1–C6) across the three repositories
- 20 migration items (M01–M20) requiring capability-by-capability transfer
- 7 open founder decisions (F01–F07) blocking architectural clarity
- 3 CR-001 amendments required before migration planning could proceed

### Seven Ambiguities Requiring Resolution

Prior to this ADR, the following were unresolved:

1. Whether the V1 Golden Path should include LeadOS/SalesOS
2. Whether CommunityOS belongs to CreatorOS or another domain
3. Whether ResearchOS belongs to CreatorOS or SOREYA/Inflow
4. Whether ContentFabrik is a module, a platform, or an internal engine name
5. Whether LeadFabrik is a module or a configured solution
6. Whether `creatoros-insight` should be archived, maintained, or merged
7. How to handle divergence between the three repositories during migration

This ADR resolves all seven through Founder Decisions F01–F07.

---

## 6. Decision

The following decisions are **binding and approved**:

1. `creatoros-platform` is the **single canonical CreatorOS repository**. All new product development occurs here only.
2. `creatoros-insight` is a **read-only migration source**. No new features. No backporting. Archived after verified migration.
3. `content-fabrik` is a **read-only ContentOS migration source and API contract reference**. Archived after unique capabilities, runtime dependencies, and contract ownership are verified.
4. The **CreatorOS V1 Golden Path is frozen**: BrandOS → ContentOS → LaunchOS → ManagementOS → AnalyticsOS.
5. LeadOS and SalesOS are **additional CreatorOS platform modules**, outside the frozen V1 Golden Path.
6. CommunityOS is an **optional CreatorOS add-on module** under LXST SYSTEMS / Outflow. Not part of the V1 Golden Path.
7. ResearchOS **belongs to SOREYA / Inflow**. Its prior CreatorOS placement is Superseded / Legacy.
8. ContentOS is the **canonical public module name**. ContentFabrik is an internal engine name only.
9. LeadFabrik is a **configured solution and pilot** — not a standalone module. Its LF governance remains authoritative.
10. Migration is **one-way** into `creatoros-platform`. No bidirectional synchronization.
11. All migrations are **capability-based** (not file-copy) and require validation gates before archival.
12. Source repository archival is **deferred** until all approved capabilities, routes, schemas, and dependencies are verified in `creatoros-platform`.

## 9. V1 Golden Path Clarification (F01)

The CreatorOS V1 Golden Path is:

```
BrandOS → ContentOS → LaunchOS → ManagementOS → AnalyticsOS
```

**This sequence is frozen.** Any future change to the Golden Path requires a new Founder Decision or ADR amendment.

LeadOS and SalesOS are additional CreatorOS platform modules and must **not** be silently inserted into the V1 Golden Path.

The `ModuleEcosystemFlow` visualization component (`creatoros-insight/src/components/modules/ModuleEcosystemFlow.tsx`) renders the five-module horizontal pipeline and is a pending migration target (CR-001 M13). Its content must reflect the frozen V1 sequence upon migration.

## 10. CommunityOS Decision (F02)

CommunityOS is an **optional CreatorOS add-on module**.

- **Operating Area:** LXST SYSTEMS
- **Platform:** CreatorOS
- **Classification:** Optional Add-on Module
- **Golden Path:** No

CommunityOS concerns the operative relationship between Creator/organisation and their audience: interactions, community relationships, engagement, and community-to-lead transitions. This is Outflow and operative execution, not SOREYA/Inflow.

The `communityos` entry in `src/config/modules.ts` reflecting `planned addon` classification is correct and should be preserved.

---

## 11. ResearchOS Decision (F03)

ResearchOS **belongs to SOREYA / Inflow**, not to CreatorOS.

- **Operating Area:** SOREYA
- **Domain:** Inflow
- **Purpose:** Research, understanding, evidence and knowledge synthesis

The prior CreatorOS placement of ResearchOS is **Superseded / Legacy**.

**Important:** CreatorOS modules may consume research outputs (e.g. market research or audience data). However, the canonical ResearchOS product identity and knowledge logic reside in SOREYA. CreatorOS must not maintain a second ResearchOS product.

Operative market or audience research required within CreatorOS is either:
- Carried as a limited function within BrandOS, or
- Obtained via integration with SOREYA/ResearchOS.

**Action required before `modules.ts` removal:** Audit all references to `researchos` across `creatoros-platform` source files to confirm no active imports or route bindings exist. Remove or mark `status: 'superseded'` only after verified zero-dependency confirmation.

| Name | Type | Status |
|---|---|---|
| ContentOS | Public module name | Canonical — use in all product-facing contexts |
| ContentFabrik | Internal engine name | Acceptable internally; must not appear as a separate public module |

- [ ] All unique capabilities have been assessed

- [ ] Builds and routes in `creatoros-platform` have been verified
- [ ] Data and schema compatibility has been confirmed
- [ ] Runtime dependencies (if any) have been resolved or replicated

This model identifier is **unverified** against current OpenAI model offerings. Resolution is required before production deployment. See Open Implementation Question I02.

**LeadFabrik is a configured product solution and pilot** — not a module.

```
LeadFabrik
├── LeadOS
├── SalesOS
├── selected ContentOS capabilities
└── selected BrandOS capabilities
```

LeadFabrik is the concrete go-to-market solution built from multiple CreatorOS modules. It is not renamed and not dissolved during platform migration.

The existing **LF governance remains authoritative**:

- Owner: Soleil Sophia Lucestella Voggenreiter
- Operating Area: LXST SYSTEMS
- Product: LeadFabrik
- Domain: Outflow

LeadFabrik is not equivalent to LeadOS. LeadOS is a module; LeadFabrik is the configured experience built on top of it.

## 15. Migration-Source Rules (F06, F07 — partial)

The following rules are **binding** for all migration work:

1. **One-way only.** Migration flows from source repositories into `creatoros-platform` exclusively.
2. **Capability-based, not file-copy.** Each migration item is assessed, adapted, and integrated — not blindly copied.
3. **Validation gates required.** Each migration phase must pass its gate before the next phase begins (see Section 19).
4. **Source repositories are read-only.** No new product features, no fixes, no refactors in `creatoros-insight` or `content-fabrik` during migration.
5. **Route basename correction required.** `creatoros-insight` uses `basename: '/'`; `creatoros-platform` uses `basename: '/platform'`. All migrated routes must be adapted to the platform basename.
6. **Storage schema compatibility required.** The v1 BrandProfile key (`creatoros-brand-profile-v1`) must remain backwards-compatible if the Extended BrandProfile (`creatoros-brand-profile-extended-v1`) is migrated. Write-through behaviour must be verified (CR-001 Amendment 3).
7. **`src/core/` handling.** The typed domain kernel (`src/core/`) is functionally identical in both platform and insight (13 subdomains). The platform version is canonical. Insight's `src/core/` is not migrated — it is already present.

---

## 16. Divergence-Management Rules (F07)

```
No bidirectional synchronization.
No new feature development in source repositories.
All migrations are capability-based and validated before archival.
```

Specific rules:

1. No new product features in `creatoros-insight` or `content-fabrik`.
2. No changes from `creatoros-platform` back into the source repositories.
3. Every migration item is tracked in the CR-001 migration matrix (M01–M20).
4. Conflicts are resolved in favour of the canonical architecture — not by file date or line count.
5. Discarded or rejected source capabilities are documented explicitly.
6. Archival of source repositories occurs only after full migration verification.
7. Post-migration, `creatoros-platform` is the **single source of truth** for all CreatorOS product development.

## 17. Consequences

### Positive

- **Single source of truth.** One canonical repository eliminates divergence, merge conflicts, and duplicate maintenance.
- **Clear module ownership.** Each module belongs to exactly one domain (CreatorOS, SOREYA, or configured solution).
- **Frozen V1 scope.** The Golden Path is stable and not subject to silent expansion.
- **Clean archival.** Source repositories are archived only after verified, gated migration — no premature disposal of capabilities.

### Negative

- **Engineering effort.** Migration of 20 capability items (M01–M20) requires time, validation, and deliberate integration work.
- **Lovable/insight development paused.** No new features may be built in `creatoros-insight` during migration, which means the Lovable-based development workflow is suspended.
- **ResearchOS separation cost.** Moving ResearchOS to SOREYA requires a reference audit and potential route/import cleanup in `creatoros-platform`.
- **Deferred archival.** Source repositories remain as active (if frozen) local folders until migration is fully verified, adding maintenance overhead.

## 19. Migration Validation Gates

Migration follows CR-001's 20-item matrix (M01–M20). The following gates must be passed before proceeding to the next phase.

### Phase Gates

| Phase | Gate Condition |
|---|---|
| Phase 0 — Prerequisites | OpenAI model identifier verified (I02); Supabase slug renamed (I01); `@replit/connectors-sdk` audited (I03); package name corrected (I10) |
| Phase 1 — Backend | Edge function `make-server-add905f8` integration verified; Supabase project URL confirmed; real OpenAI call tested in staging |
| Phase 2 — Core BrandOS | BrandProfile v1 read/write verified in platform; Extended BrandProfile write-through to v1 key confirmed before insight's 780-line version replaces platform's 413-line version (Amendment 3) |
| Phase 3 — Core ContentOS | ContentOS production workflow wired via `src/core/` pipeline; content library storage verified |
| Phase 4 — Module Expansion | LaunchOS (M09), ManagementOS (M10), AnalyticsOS (M11) migrated, routes adapted to `/platform` basename; diagnostics module (M12) integrated |
| Phase 5 — UI / Shell | ModuleEcosystemFlow (M13), DashboardOverview (M14), HeroSection (M15) migrated; all 9 localStorage keys validated (diagnostics gate) |
| Phase 6 — Routes | All routes from `creatoros-insight` assessed; accepted routes adapted with correct basename; HandoffPage CTA verified (I06) |
| Phase 7 — Archival | `creatoros-insight` archival checklist complete; `content-fabrik` archival checklist complete; zero regressions on V1 Golden Path |

### Per-Capability Gates (CR-001 Amendments Applied)

- **Amendment 1:** `content-fabrik` archival gate requires verified capability migration AND runtime dependency validation — defer/archive later, not on schedule.
- **Amendment 2:** Phase 0 is blocked until `"gpt-5-mini"` is verified against current OpenAI model offerings.
- **Amendment 3:** Phase 2 BrandOS gate requires write-through verification before the Extended BrandProfile page replaces the platform's current implementation.
